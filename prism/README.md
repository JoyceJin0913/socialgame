---

## 这个目录是什么

`prism/` 目录是一份**剧情多样性方法论的提案 + 应用样板**，由协作方贡献，等待你 review。

它**不修改你现有的任何代码**。`dist/` 下的 index.html、play 模块、chat.js 都没动过。这只是一份独立目录的设计稿。

---

## 它解决的是什么问题

我读了 `dist/play/index.html`，发现一个事：

当前的第五幕三场（一年之约 / 凯旋归来 / 一纸休书）每一场只有 3 个固定选项，**前面的对话选择不影响后面**。真正决定结局的，只有最后的麦克风 QTE。这意味着前三场对话本质上是装饰——玩家二周目会立刻发现"反正都一样"。

Prism 的目标是给这一段**加上结构化的多样性**，让"在不同状态下进入同一个场景"会看到不同的选项。

---

## 它和你现有系统的关系

**Prism 不取代你的任何东西。** 你的角色系统、对话系统、QTE、Serverless 调用、UI——全部保留。

Prism 只在**节点定义层**加一层薄薄的结构：把"一个场景 = 3 个固定选项"改成"一个场景 = 1 个 Crystal + 1 个候选选项池 + 当前世界状态"。

具体边界：

| 层次 | 谁负责 |
| --- | --- |
| 节点的戏剧骨架、选项池、可见性规则 | Prism（这个目录） |
| 数值系统、累积、阈值、结局判定 | 你已有的或后续要加的数值层 |
| AI persona、对话生成、QTE | 你现有的 chat.js + play/index.html |
| 部署、UI、用户旅程 | 你现有的 dist/ |

## 核心想法（30 秒版本）

**节点不是一段写好的内容，是一个生成器函数。**

作者只写两件事：
1. 这一幕**必须发生什么**（戏剧功能）
2. 一份**候选选项池**

至于"哪些选项现在能浮出来、代价多大"——由当前世界状态算出来。

世界被分解成 5 条独立的轴：

| 轴 | 管什么 |
| --- | --- |
| **S** Scene | 事件发生在哪 |
| **H** Hidden | 玩家知道多少真相 |
| **N** NPC | 谁在场 |
| **T** Time | 还剩多少时间 |
| **A** Ambient | 舆论 / 天气等环境反响 |

## 我具体做了什么

针对**你 repo 里现有的第五幕三场**，我做了一份双视角重构样板（在 `design-doc.md`）：

- 寒雁视角：3 个场，每场 1 个 Crystal + 3–5 个候选选项
- 傅云夕视角：同 3 个场，但 H 轴默认值不同，选项集完全不同

每个选项带一个 `hook` 字段（语义事件名），用来跟你的数值系统对接。

`demo.html` 里的交互折射台用的就是这份重构素材，可以现场拖动五轴看选项怎么变化。

## 跟你现有代码的对接思路（最小改动版）

如果你将来想接入 Prism，最小改动路径是**扩展现有的 STORY 结构**：

**现状**（`dist/play/index.html`）：
```js
scenes: [
  {
    id: 'farewell',
    aiOpening: '...',
    options: [
      { text: '...', tag: '...' },
      { text: '...', tag: '...' },
      { text: '...', tag: '...' }
    ]
  }
]
```

**接入 Prism 后**：
```js
scenes: [
  {
    id: 'farewell',
    crystal: {
      mustHappen: ['男主下达一年之约'],
      mustNotHappen: ['当面坦白寒毒']
    },
    axes: {
      S: 'study',
      H: { illness: 'unknown' },
      N: ['fyx', 'maid'],
      T: 20,
      A: { rumor: 'private' }
    },
    choicePool: [        // ← 候选选项池，比 options 多
      { text: '...', hook: 'vow_loyalty', require: {} },
      { text: '...', hook: 'probe_intent', require: {} },
      { text: '...', hook: 'silent_consent', require: {} },
      { text: '...', hook: 'investigate_illness',
        require: { 'H.illness': 'hinted_or_above' } }   // ← 信息差驱动的新选项
    ]
  }
]
```

然后在 `chat.js` 或一个新文件里加一个 `filterChoices(scene, currentAxes)` 函数，根据当前轴过滤可见选项。

**这是建议，不是要求。** 如果你觉得这个改造方向不对，或者你已经有别的想法，告诉我。

## 几个你可能想问的问题

### Q1 · 这跟我现有的数值系统冲突吗？
**不冲突。** Prism 完全不动数值。选项的 `hook` 是个语义事件名（比如 `vow_loyalty`），数值系统自己决定这个事件对应加多少分、扣多少声望。两层独立迭代。

### Q2 · 我必须用所有 5 个轴吗？
**不必。** 一个 Crystal 可以只用 2 个轴（比如只调 S 和 H），其他三个固定。轴是工具不是义务。

### Q3 · 这对 AI 对话生成有影响吗？
**有一点，方向是正向的。** 当前的 system prompt 是按场景写死的；如果引入 Prism，可以把"当前 axes 状态"加进 prompt 上下文，AI 会知道"现在是雪夜书房 / 没人在 / 隐疾未知"，对话会更贴合具体世界。这部分我没改 chat.js，留给你决定要不要做。

### Q4 · 五幕的其他幕怎么办？
**先用第五幕跑通流程。** 跑通后你（或我）可以照着 design-doc 的模式扩展到其他幕。这是个增量工作，不需要一口气全改。

### Q5 · 多人 / Social 场景能用吗？
**能，但 Prism 不替代你的 social 机制。** 比如你的投票、联盟、密聊这些社交机制不属于 Prism 管的事。Prism 只管单个节点内"玩家此刻能看到什么选项"。多人状态下，每个玩家有自己的 axes 视图，但这是工程实现的事。

### Q6 · 我可以怎么改 / 拒绝 / 局部采纳？
**完全自由。** 这是一份提案。可能的处理方式：
- **全部采纳**：把 design-doc 里的样板真的接入 STORY，跑通第五幕
- **部分采纳**：只接受方法论，自己改造
- **暂存观望**：保留这个目录作为方法论参考，先不动主代码
- **拒绝合并**：我把这个分支留在远端，作为未合并的研究分支
